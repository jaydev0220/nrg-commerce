# Release Verification

Run these checks from an approved operator host. Do not run them from a GitHub-hosted runner or an untrusted network. Keep the output with the release record.

## Capacity test

The capacity runner exercises these workloads concurrently during the steady phase:

- 100 storefront reads per second, with a 200-per-second burst;
- 20 authenticated staff sessions, paced to one dashboard read per second per session;
- 5 accepted inquiries per second, with a 10-per-second burst.

It fails when the configured availability, p95, p99, request-count, or achieved-rate threshold is missed. The latency thresholds match the service-level objectives.

Before testing:

1. Deploy the exact release commit to a protected staging environment.
2. Point the staging contact recipient at a controlled delivery sink.
3. The staging contact Worker uses a dedicated 1,000-request-per-minute binding so this test can run without changing production protection. Ensure the single approved load-generator IP is not constrained below the target by additional staging edge or API limits. Restore any temporary edge allowance immediately after the run. Production limits must still be validated separately with multiple source IPs.
4. Create `.release-secrets` with mode `0700`.
5. Create 20 active authenticated sessions. Put one `__Host-admin_access_token=VALUE` cookie header on each line of `.release-secrets/staff-cookies.txt`.
6. Set every secret file to mode `0600`. Do not include refresh tokens or use symbolic links.

For normal Turnstile credentials, place one fresh, unique token per inquiry request in `.release-secrets/turnstile.tokens`. At the default duration, 400 tokens are required. Tokens expire after five minutes and are single-use, so generate them immediately before the run.

For automated staging, use Cloudflare's testing sitekey and testing secret, a staging-only recipient, and the dummy token generated by the test widget. Never place test credentials in production. The official behavior is documented in [Cloudflare Turnstile testing](https://developers.cloudflare.com/turnstile/troubleshooting/testing/).

Create `.release-secrets/capacity.env` with mode `0600`:

```dotenv
CAPACITY_API_BASE_URL=https://api.staging.example.com
CAPACITY_CONTACT_URL=https://contact.staging.example.com/inquiry
CAPACITY_REQUEST_ORIGIN=https://catalog.staging.example.com
CAPACITY_CONFIRMED_API_ORIGIN=https://api.staging.example.com
CAPACITY_CONFIRMED_CONTACT_ORIGIN=https://contact.staging.example.com
CAPACITY_STAFF_COOKIES_FILE=.release-secrets/staff-cookies.txt
CAPACITY_INQUIRY_EMAIL=delivery-sink@example.com
CAPACITY_TURNSTILE_TOKENS_FILE=.release-secrets/turnstile.tokens
```

For the staging test secret, replace `CAPACITY_TURNSTILE_TOKENS_FILE` with:

```dotenv
CAPACITY_TURNSTILE_TEST_TOKEN=XXXX.DUMMY.TOKEN.XXXX
CAPACITY_CONFIRM_TURNSTILE_TEST_MODE=staging-test-secret
```

Optional overrides are `CAPACITY_DURATION_SECONDS`, `CAPACITY_BURST_SECONDS`, `CAPACITY_REQUEST_TIMEOUT_SECONDS`, `CAPACITY_STOREFRONT_RATE`, `CAPACITY_STAFF_CONCURRENCY`, `CAPACITY_STAFF_INTERVAL_MS`, and `CAPACITY_INQUIRY_RATE`.

Run:

```bash
node --env-file=.release-secrets/capacity.env scripts/operations/verify-capacity.mjs
```

Every row must report `PASS`. Delete the session and token files after recording the non-sensitive output.

## Post-deploy verification

This verifier sends exactly one password login, one authenticator verification, one storefront list request, one product-detail request, one ranged image request, one inquiry, and one logout. It also checks liveness and database readiness. The staff account must already have authenticator MFA configured.

Generate a fresh Turnstile token for the `inquiry` action immediately before running the command. Production tokens are single-use; the verifier never retries the inquiry.

Create `.release-secrets/verify.env` with mode `0600`:

```dotenv
VERIFY_API_BASE_URL=https://api.example.com
VERIFY_CONTACT_URL=https://contact.example.com/inquiry
VERIFY_CDN_ORIGIN=https://cdn.example.com
VERIFY_API_ORIGIN=https://admin.example.com
VERIFY_CONTACT_ORIGIN=https://catalog.example.com
VERIFY_CONFIRMED_API_ORIGIN=https://api.example.com
VERIFY_CONFIRMED_CONTACT_ORIGIN=https://contact.example.com
VERIFY_CONFIRMED_CDN_ORIGIN=https://cdn.example.com
VERIFY_STAFF_EMAIL=operator@example.com
VERIFY_STAFF_PASSWORD=replace-at-runtime
VERIFY_TOTP_CODE=123456
VERIFY_INQUIRY_EMAIL=delivery-sink@example.com
VERIFY_TURNSTILE_TOKEN=replace-at-runtime
```

Run:

```bash
node --env-file=.release-secrets/verify.env scripts/operations/verify-deployment.mjs
```

All eight checks must report `PASS`. Confirm the inquiry reached the delivery sink, then securely delete `verify.env`. If a check fails after authentication, the verifier still attempts logout and preserves the original failure message.

## Release evidence

Record:

- release commit SHA and immutable API image digest;
- staging and production Worker version IDs;
- migration deployment and zero-drift results;
- capacity-test output and operator source IP;
- post-deploy verifier output and inquiry delivery confirmation;
- latest backup object, checksum, and restore-drill date;
- dashboard links, paging test result, approver, and rollback decision.
